  <files>services/daily_pipeline.py</files>
  <action>
    Read services/daily_pipeline.py and services/scheduler.py.

    The _log_run() function calls queries.insert_pipeline_run() which is now async (Phase 8 migration), but run_daily_pipeline is a sync function running in APScheduler's thread pool.

    Fix: Convert _log_run to use asyncio.run() to call the async query, OR wrap the entire pipeline runner to be called via an async scheduler.

    The simplest safe fix: in _log_run(), use asyncio.run(queries.insert_pipeline_run(conn, ...)) if there's no running event loop, or asyncio.get_event_loop().run_until_complete() if there is one. Since APScheduler BackgroundScheduler runs in threads, asyncio.run() is the correct approach.

    After fix, verify: python -c "from services.daily_pipeline import run_daily_pipeline; print('OK')"
  </action>
  <done>No RuntimeError on import; _log_run correctly calls the async query function.</done>

    ```python
        "macropulse_pipeline_runs_total",
        "Total pipeline runs",
        ["status"],  # labels: success, failure
    )

        "macropulse_pipeline_duration_seconds",
        "Pipeline run duration in seconds",

        "Unix timestamp of last successful pipeline run",
    )

        "macropulse_db_pool_size",
        "Total asyncpg connection pool size",
    )

    DB_POOL_IDLE = Gauge(
        "macropulse_db_pool_idle",
        "Idle asyncpg connection pool connections",
    )
    ```

    All metrics are module-level singletons to prevent ValueError: Duplicated timeseries on import.
